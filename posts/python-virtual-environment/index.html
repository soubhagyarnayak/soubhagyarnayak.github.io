Python Virtual Environment

2018-08-23
1 minute read

Virtual environments help to have different applications depending upon 2 different versions of a same dependency to be usable on the same machine.

As an illustration, let us say App1 has a dependency on version 1 of lib1. If we are installing the application App1 on our machine, then by default the site-packages will contain version 1 of lib1. Now if the user wants to use App2 which has a dependency on version 2 of lib1 then we are in trouble. Updating lib1 may break App1. Similarly, during development, we may not want to pollute our python installation with numerous library dependencies.

Virtual environments solve this problem by simulating a virtual python environment for an application. Now if App1 is associated with virtual environment env1 and App2 is associated with virtual environment env2, then we need not worry for version mismatch of common dependencies.

We have 2 tools to support virtual environments in Python. virtualenv needs to be installed separately and venv comes as part of Python3. So, I will focus on venv in this article, even though virtualenv has the same capabilities.

On *NIX platform, the following commands are used to interact with venv.

1#create virtual environment 
2python3 -m venv <PathToVirtualEnvDir> 
3#use virtual environment source 
4<PathToVirtualEnvDir>/bin/activate 
5#get out of virtual environment 
6deactivate

For windows platform, the following commands are used to interact with venv.

1#create virtual environment 
2python -m venv <PathToVirtualEnvDir> 
3#use virtual environment 
4<PathToVirtualEnvDir>\Scripts\activate 
5#get out of virtual environment 
6deactivate
